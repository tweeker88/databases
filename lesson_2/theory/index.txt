Уровни проектирования БД:
- Концептуальная модель
- Логическая модель
- Физическая модель

Концептуальная модель:
- Определение сущностей и их документирование
- Определение связей между сущностями
- Создание ER модели
- Определение атрибутов
- Определение потенциальных ключей

Логическая модель:
- Выбор модели данных
- Определение набора таблиц
- Декомпозиция
- Нормализация данных
- Денормализация данных
- Определение набора транзакций и соответствие структуры данных
- Определение требований поддержки целостности

Физическая модель:
- Проектирование таблиц средствами СУБД
- Реализация бизнес правил
- Определние транзакционной модели
- Проектирование физ.организации данных
- Планирование ресурсов
- Определние правил безопасности
- Организация сопровождения и мониторинга
- Выбор структуры хранения
- Оценка нагрузки, роста, производительности
- Определение требований к производительности
- Организация резервного копирования
- Организация CI/CD
- Обеспечение высокой доступности

ACID:
Atomicity - транзакции атомарны
Consistency - транзации не нарушают согласованность данных, то есть они переводят БД из одного корректного состояния в другое
Isolation - работая одновременно транзакции не влияют друг на друга
Durability - если транзакция была завершена, никакое внешнее событие не должно привести к потере совершенных ей изменений.

MVCC:
- Механизм, обеспечиваюий параллельного доступа к БД
- Каждой сессии предоставляется "снимок" БД
- Изменения в БД невидимы другим пользователям до момента фиксаций транзакций
- читатели не блокируют писателей
- писатели не блокируют читателей
- читатели не блокируют читатей

OLTP - Online Transaction Processing
- Оперирует текущими данными
- Большой процент операций записи
- Требуется большая скорость обработки транзакций
- Много конкурентных сессий
- Требуется консистентость данных
Требования:
- Время выполнения транзакций
- Конкурентная загрузка
- Минимальная время простоя

DWH - Data Warehouse
- Часто денормализована
- Длинные транзакции
- Используется для аналитики
- Требует много индексов
- Используется технология для построения аналитических отчетов

Репликация:
- Жерналы (REDO logs, WAL, transactions logs)
- Процесс синхронизации данных путем доставки журналов на другие ноды
- Задачи: балансировка чтения и записи (fault tolerance)
- Проблемы: организация автоматического переключения
- Разновидность: Асинхрон, синх, полусинх.

Кластеризация:
- Кластеризация: полноценная масштабируемая, отказоустойчивая
- MySQL: Galera cluster, InnoDB cluster
- PostgreSQL: Patroni, Slony
- Redis cluster

Proxy, Pool, Balancer:
- Экономия времени на коннект к БД
- Поддержка очередей
- Ограничение кол-ва подключения
- Балансировка запросов на чтение и запись между нодами
- Балансировка запросов на запись в разные БД на разные ноды в режиме мульти-кластер
- Анализ запросов

Резервное копирование:
- Полное
- Инкрементальное
- Дифференциальное
